# 测试说明

## 1. 测试环境准备
- 确保已完成《部署文档》中的 Docker 部署，三个容器（db / backend / gateway）均处于运行状态。
- 准备至少两份 PDF 文件：
  - **法律文档**：用于知识库上传（如某法规全文、合同范本条款等）。
  - **待审合同**：用于合同审核接口（可含故意设置的违规条款便于验证）。

## 2. 知识库上传接口测试

### 2.1 测试目的
验证 PDF 解析、分块、向量化及写入 pgvector 的完整流程。

### 2.2 请求示例 (cURL)
```bash
# 将 <法律文档.pdf> 替换为实际路径
curl -X POST "http://localhost/api/knowledge/upload" \
  -H "accept: application/json" \
  -H "Content-Type: multipart/form-data" \
  -F "file=@/path/to/法律文档.pdf"
```

### 2.3 预期结果
- **状态码**：200
- **响应体**：`{"status":"success","doc_id":1,"message":"成功入库 N 个知识块"}`（N 为解析出的文本块数量）。

## 3. 合同审核接口测试

### 3.1 测试目的
验证合同上传 → RAG 检索 → LLM 审核 → 结构化结果返回的完整链路。

### 3.2 请求示例 (cURL)
```bash
curl -X POST "http://localhost/api/contract/audit" \
  -H "accept: application/json" \
  -H "Content-Type: multipart/form-data" \
  -F "file=@/path/to/待审合同.pdf"
```

### 3.3 预期结果
- **状态码**：200
- **响应体**：JSON 结构包含 `status`（PASS/FAIL）、`risk_level`、`violations`（数组）、`summary`。若 LLM 返回非标准 JSON，可能包含 `error` 与 `raw_output` 字段。

## 4. 前端功能测试步骤

| 步骤 | 操作 | 预期结果 |
|------|------|----------|
| 1 | 浏览器访问 `http://<服务器IP>` | 进入系统首页，左侧可见「合同智能审核」「法律知识库管理」菜单 |
| 2 | 进入「法律知识库管理」，选择 PDF 并点击「开始解析并入库」 | 页面提示成功，并显示入库知识块数量 |
| 3 | 进入「合同智能审核」，上传合同 PDF 并点击「开始智能审核」 | 右侧展示审核报告，包含风险等级、违规点及建议 |

## 5. 常见失败场景与排查
- **上传后报 500**：查看 `docker logs reviewer_backend`，常见原因为 OPENAI_API_KEY 未配置或 PDF 解析异常。
- **审核一直 loading 无结果**：检查后端日志是否出现 LLM 超时或 API 限流；可适当调大 Nginx 的 `proxy_read_timeout`。
- **知识库上传成功但审核时无相关法条**：确认先完成知识库文档上传，且合同内容与已入库法规在语义上有关联，否则 RAG 召回的参考可能为空或相关性低。
