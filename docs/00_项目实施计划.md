# AI合同智能审核系统 - 项目实施计划

## 1. 需求拆解
- **知识库管理模块**：
  - 输入：法律法规PDF文件。
  - 输出：结构化法条文本、向量数据存入pgvector。
  - 验收标准：PDF解析准确率高，向量化成功，可按相似度检索。
- **合同审核模块（RAG+LLM）**：
  - 输入：待审核合同PDF文件。
  - 输出：包含风险等级、违规点、依据、建议的审核报告JSON。
  - 验收标准：检索到相关法条，LLM准确识别合同中的违规项并给出合理建议。
- **前端展示模块**：
  - 输入：用户交互（上传、查看结果）。
  - 输出：友好的可视化界面。
  - 验收标准：页面响应快，交互清晰，PC端适配良好，支持展示复杂审核结果。
- **系统基础能力**：
  - 输入/输出：登录鉴权、接口请求限流、操作日志记录。
  - 验收标准：非授权无法访问核心接口，高并发有基础防护，操作有迹可循。

## 2. 技术架构细化
- **前端**：Vue 3 + Vite。负责UI展示与交互封装。
- **网关/反向代理**：Nginx。负责前端静态资源托管、API请求转发、基础接口限流。
- **后端**：FastAPI (Python)。负责核心业务逻辑：PDF解析、文本分块(Chunking)、调用Embedding模型生成向量、RAG检索、组装Prompt调用LLM。
- **数据库**：PostgreSQL + pgvector插件。负责存储业务关系数据及法条文本的高维向量。
- **大模型**：外接OpenAI风格API（例如GPT-4或兼容OpenAI格式的本地/云端模型）。

## 4. 数据库设计规划
- `users`: 用户鉴权与角色。
- `knowledge_docs`: 记录上传的法律法规文档元数据。
- `knowledge_chunks`: 存储法律文档的分块文本内容及对应向量（`embedding`字段，使用pgvector的`vector`类型）。
- `audit_records`: 记录合同审核任务及返回的结构化结果。

## 5. 风险预案
- **PDF解析精度较差**：预案 - 引入专门针对排版的解析库（如 `pdfplumber`）并清洗特殊字符。
- **RAG检索不精准导致审核幻觉**：预案 - 调整分块策略（Chunk size/overlap），在Prompt中强制要求LLM“仅基于提供的参考法条进行审核，若无相关依据则判定为无风险”。
- **LLM API调用超时/失败**：预案 - 增加请求重试机制，对超长文档考虑分段审核。

## 6. 部署方案
采用 Docker 容器化方案：
- 构建前端镜像（基于Nginx，包含构建后的dist文件及Nginx配置）。
- 构建后端镜像（基于Python 3.10+，安装相关依赖，通过Uvicorn启动）。
- 数据库使用官方 `pgvector/pgvector:pg15` 镜像。
- 通过 `docker-compose.yml` 统一编排网络与数据卷。
