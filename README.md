# AI合同智能审核系统 - 项目实施计划

## 1. 需求拆解
- **知识库管理模块**：
  - 输入：法律法规PDF文件。
  - 输出：结构化法条文本、向量数据存入pgvector。
  - 验收标准：PDF解析准确率高，向量化成功，可按相似度检索。
- **合同审核模块（RAG+LLM）**：
  - 输入：待审核合同PDF文件。
  - 输出：包含风险等级、违规点、依据、建议的审核报告JSON。
  - 验收标准：检索到相关法条，LLM准确识别合同中的违规项并给出合理建议。
- **前端展示模块**：
  - 输入：用户交互（上传、查看结果）。
  - 输出：友好的可视化界面。
  - 验收标准：页面响应快，交互清晰，PC端适配良好，支持展示复杂审核结果。
- **系统基础能力**：
  - 输入/输出：登录鉴权、接口请求限流、操作日志记录。
  - 验收标准：非授权无法访问核心接口，高并发有基础防护，操作有迹可循。

## 2. 技术架构细化
- **前端**：Vue 3 + Vite。负责UI展示与交互封装。
- **网关/反向代理**：Nginx。负责前端静态资源托管、API请求转发、基础接口限流。
- **后端**：FastAPI (Python)。负责核心业务逻辑：PDF解析、文本分块(Chunking)、调用Embedding模型生成向量、RAG检索、组装Prompt调用LLM。
- **数据库**：PostgreSQL + pgvector插件。负责存储业务关系数据及法条文本的高维向量。
- **大模型**：外接OpenAI风格API（例如GPT-4或兼容OpenAI格式的本地/云端模型）。

## 4. 数据库设计规划
- `users`: 用户鉴权与角色。
- `knowledge_docs`: 记录上传的法律法规文档元数据。
- `knowledge_chunks`: 存储法律文档的分块文本内容及对应向量（`embedding`字段，使用pgvector的`vector`类型）。
- `audit_records`: 记录合同审核任务及返回的结构化结果。

## 5. 风险预案
- **PDF解析精度较差**：预案 - 引入专门针对排版的解析库（如 `pdfplumber`）并清洗特殊字符。
- **RAG检索不精准导致审核幻觉**：预案 - 调整分块策略（Chunk size/overlap），在Prompt中强制要求LLM“仅基于提供的参考法条进行审核，若无相关依据则判定为无风险”。
- **LLM API调用超时/失败**：预案 - 增加请求重试机制，对超长文档考虑分段审核。

## 6. 部署方案
采用 Docker 容器化方案：
- 构建前端镜像（基于Nginx，包含构建后的dist文件及Nginx配置）。
- 构建后端镜像（基于Python 3.10+，安装相关依赖，通过Uvicorn启动）。
- 数据库使用官方 `pgvector/pgvector:pg15` 镜像。
- 通过 `docker-compose.yml` 统一编排网络与数据卷。

# 项目总览

## 1. 项目背景
在企业法务和合同管理场景中，传统人工审核合同耗时长、易漏看合规风险。本项目旨在利用大语言模型（LLM）的理解推理能力，结合检索增强生成（RAG）技术，打造一个自动化的「AI合同智能审核系统」。系统通过检索内部积累的法律法规知识库，自动对比合同条款，输出客观、准确的审核意见。

## 2. 核心功能
1. **法律知识库管理**：支持上传PDF法律法规文档，系统自动提取文本、分块并进行向量化，存入专属向量数据库。
2. **合同智能审核**：上传待审合同后，系统从知识库召回相关法条，并让大模型基于法条对合同进行合规审查。
3. **审核结果结构化呈现**：输出具体违规点、风险等级、违规依据及修正建议。
4. **系统基础支撑**：包含简单的用户鉴权机制、操作记录以及系统级的并发限流保护。

## 3. 技术架构
- **前端 (Vue 3)**：提供可视化的文档上传、知识管理与结果查看面板。
- **网关 (Nginx)**：统一入口，分发静态资源与API请求，配置请求频率限制。
- **后端 (FastAPI)**：处理核心业务流（鉴权、文件处理、RAG调度）。
- **数据库 (PostgreSQL + pgvector)**：业务数据与向量数据的统一存储，避免引入复杂的单一向量库组件。
- **AI 引擎**：兼容 OpenAI API 格式的大模型服务，用于文本 Embedding 及对话生成。

## 4. 数据流转
1. **知识入库流**：`前端PDF上传 -> Nginx转发 -> FastAPI解析PDF -> 分块(Chunking) -> 调用Embedding模型 -> pgvector入库`。
2. **合同审核流**：`前端合同上传 -> FastAPI解析文本 -> 调用Embedding模型生成合同向量 -> pgvector相似度检索召回法条 -> 组装Prompt -> 调用LLM生成审核报告 -> 结果存入DB -> 返回前端展示`。

## 5. 部署流程概述
系统基于 Docker 进行容器化部署。只需准备好装有 Docker 和 Docker Compose 的服务器，在项目根目录复制 `.env.example` 为 `.env` 并填写 `OPENAI_API_KEY` 等环境变量，执行 `docker-compose up -d --build` 或 `./scripts/start.sh` 即可一键拉起包括 Nginx 网关、前端静态、FastAPI 后端、PostgreSQL（pgvector）在内的完整服务。详见《部署文档》。

## 6. 常见问题排查
- **无法连接数据库**：检查 PostgreSQL 容器是否正常启动，以及 `backend` 容器内的 `DATABASE_URL` 环境变量配置是否正确。
- **LLM 审核报错/超时**：查看后端日志，确认 `OPENAI_API_KEY` 及 `OPENAI_API_BASE` 是否有效，网络是否能够访问对应的 API 地址。
- **前端跨域问题**：检查 Nginx 配置文件中的反向代理设置，确保 API 请求已正确代理到后端服务，无需在后端单独处理复杂的 CORS。
